⚙️ Governance Rules — Apply at Every Decision Point
Any full initial ingest from Kapua requires a formal change request with network window approval
Dev and SysTest default to synthetic data unless ingestion pipeline testing is explicitly scoped
First-time full ingest per source-env pairing must be snapshotted and registered in the Seed Registry
Bronze clone refresh is a deliberate sprint event — never automated or self-service
Why the Seed Registry Exists

Each EDP environment (UAT, SysTest, Prod) can point at a different source system environment in Kapua — JDE Prod, JDE UAT, JDE SysTest are separate databases. Every unique EDP env × Source env pairing historically required a full re-ingestion across the 200Mb ExpressRoute link. The Seed Registry eliminates repeat network cost: the first ingest per pairing is snapshotted into ADLS, and all future resets restore from the snapshot — zero Kapua access, zero ExpressRoute load.

Seed Snapshot Lifecycle — Per Pairing
1
New Pairing Detected — Formal Change Request Raised

EDP team or project squad identifies a new EDP env × Source env pairing. Change request raised with network team specifying expected data volume and preferred extraction window (11pm–5am NZST). Pairing added to registry with status PENDING_INGEST.

2
Throttled Initial Ingest — Off-Peak Window

Custom MSSQL Source (or appropriate connector) runs against the target source environment. Extraction chunked by batch size and rate-limited in Python. Parquet files land in the UC Volume. DLT pipeline runs to populate Bronze Streaming Tables in the target EDP catalog. For large sources (JDE transactional history): consider Sydney DC NFS staging to decouple extraction rate from ExpressRoute throughput.

3
Snapshot Written to Seed Storage — DEEP CLONE

Once Bronze tables are fully populated and validated, a Delta deep clone is written to seed storage:

abfss://seeds@<account>.dfs.core.windows.net/<source_system>/<source_env>/<snapshot_date>/

DEEP CLONE copies the actual Parquet files — self-contained, independent of the live Bronze table. Registry status updated to SEEDED.

4
Future Resets — Restore from Seed, Not from Kapua

When the same pairing is needed again, a restore job runs:

CREATE OR REPLACE TABLE <edp_env>_catalog.bronze.<table> SHALLOW CLONE delta.`abfss://seeds/…/<table>/`

Zero ExpressRoute traffic. Zero Kapua access. Seconds per table. DLT Silver/Gold full refresh follows.

5
Seed Refresh Policy — When to Re-Seed

Seeds are point-in-time baselines — not kept current. A seed is refreshed only when: (a) source schema changes make the snapshot incompatible, (b) a project explicitly requires a more recent baseline, or (c) the seed exceeds the agreed maximum age (e.g. 6 months). Re-seeding follows the same formal change request process as the original ingest.

6
Registry Maintained as UC Table

All pairings, seed locations, snapshot dates, schema versions, and restore history tracked in governance.seed_registry.env_source_pairings. UC lineage links seed tables back to source Bronze. Admins query seed age, restore count, and storage cost per pairing from this table.

Append-only log of every restore event: who triggered it, which pairing, snapshot version, timestamp, Silver/Gold refresh job IDs downstream.
⚙️ Seed Registry Governance Rules
Full ingest from Kapua occurs only once per EDP × Source pairing — all future resets use the seed
Seeds are DEEP clones — self-contained, not dependent on live Bronze table state
Re-seeding requires the same formal change request as the original ingest
Seed restores are gated — require platform admin approval, logged in restore_history
No EDP pipeline has direct write access to the seed container
Schema version tracked per seed — incompatible seeds trigger mandatory re-seed, not silent restore
